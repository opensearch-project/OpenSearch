# Peer Recovery Gaps Deep-Dive for DFA Indices

## GAP 1: `Lucene.cleanLuceneIndex` in corruption path

### Code Traced

**Method** (`Lucene.java:253`):
```java
public static void cleanLuceneIndex(Directory directory) throws IOException {
    try (Lock writeLock = directory.obtainLock(IndexWriter.WRITE_LOCK_NAME)) {
        for (final String file : directory.listAll()) {
            if (file.startsWith(IndexFileNames.SEGMENTS)) {
                directory.deleteFile(file); // remove all segment_N files
            }
        }
    }
    try (IndexWriter writer = new IndexWriter(directory,
        new IndexWriterConfig(Lucene.STANDARD_ANALYZER)
            .setSoftDeletesField(Lucene.SOFT_DELETES_FIELD)
            .setMergePolicy(NoMergePolicy.INSTANCE)
            .setCommitOnClose(false)
            .setOpenMode(IndexWriterConfig.OpenMode.CREATE)
    )) {
        // IndexFileDeleter removes all pending files
    }
}
```

**Caller** (`RecoveryTarget.java:424`):
```java
Lucene.cleanLuceneIndex(store.directory()); // clean up and delete all files
```

### Analysis

1. `store.directory()` returns `StoreDirectory` → `ByteSizeCachingDirectory` → `DataFormatAwareStoreDirectory` → `SubdirectoryAwareDirectory` → FSDirectory.

2. **First loop**: `directory.listAll()` calls `DataFormatAwareStoreDirectory.listAll()` which returns format-prefixed names (e.g., `"parquet/_0.pqt"`, `"_0.cfs"`). The loop only deletes files starting with `"segments"` — format-prefixed files like `"parquet/_0.pqt"` don't match, so they're skipped here. `"segments_N"` (not prefixed since it's lucene format) IS deleted correctly.

3. **IndexWriter with CREATE**: Lucene's internal `IndexFileDeleter` calls `directory.listAll()`, gets format-prefixed names, and treats unrecognized files as unreferenced. It calls `directory.deleteFile("parquet/_0.pqt")` for each.

4. **deleteFile routing**: `DataFormatAwareStoreDirectory.deleteFile("parquet/_0.pqt")` → `resolveFileName("parquet/_0.pqt")` → `SubdirectoryAwareDirectory.deleteFile("parquet/_0.pqt")` → `parseFilePath("parquet/_0.pqt")` → resolves to `<shard>/parquet/_0.pqt` absolute path → `super.deleteFile(absolutePath)`.

5. **Intent check**: This code runs in the corruption catch block. The intent IS to delete everything. Format-prefixed files being deleted is correct behavior.

### VERDICT: FALSE ALARM

`DataFormatAwareStoreDirectory` correctly routes `deleteFile` for format-prefixed names. The corruption path intentionally cleans all files, and the directory chain handles this correctly. Lucene's `IndexFileDeleter` will attempt to delete format-prefixed files, and those deletions will succeed through the proper routing.

Files that should be kept are found in `sourceMetadata`; files that should be deleted are correctly routed through the directory chain.

## GAP 4: `syncSegmentsFromRemoteSegmentStore` after peer recovery

### Code Traced

**Call sites** (in order during peer recovery):

1. **Pre-recovery** (`PeerRecoveryTargetService.java:255`):
   ```java

       try {
           indexShard.syncSegmentsFromRemoteSegmentStore(false, recoveryTarget::setLastAccessTime);
       } catch (Exception e) {
           logger.error("Exception while downloading segment files from remote store, will continue...", e);
       }
   }
   ```

2. **Post-recovery engine open** (`IndexShard.java:3141`):
   ```java
   if (syncFromRemote) {
       syncSegmentsFromRemoteSegmentStore(false);
   }
   ```
   Called from `RecoveryTarget.finalizeRecovery()` → `openEngineAndSkipTranslogRecovery()` → `innerOpenEngineAndTranslog(syncFromRemote=true)`.

### `syncSegmentsFromRemoteSegmentStore(false)` behavior:

```java

Map<String, UploadedSegmentMetadata> uploadedSegments = remoteDirectory.getSegmentsUploadedToRemoteStore()
    .entrySet().stream()
    .filter(entry -> !entry.getKey().startsWith(IndexFileNames.SEGMENTS))
    .collect(...);

```

### `copySegmentFiles` skip logic:
```java
for (String file : uploadedSegments.keySet()) {
    long checksum = Long.parseLong(uploadedSegments.get(file).getChecksum());
    if (overrideLocal || localDirectoryContains(storeDirectory, file, checksum) == false) {
```

### `localDirectoryContains` for DFA:
```java
try (IndexInput indexInput = localDirectory.openInput(file, IOContext.READONCE)) {
    if (indexSettings.isPluggableDataFormatEnabled()) {
        DataFormatAwareStoreDirectory dfasd = DataFormatAwareStoreDirectory.unwrap(localDirectory);
        if (dfasd != null) {
            localChecksum = Long.parseLong(dfasd.calculateUploadChecksum(file));
        } else {
            localChecksum = CodecUtil.retrieveChecksum(indexInput);
        }
    }
    if (checksum == localChecksum) return true;
}
```

### Ordering Analysis

**Pre-recovery call (PeerRecoveryTargetService:255)**:
- Runs BEFORE peer recovery phase1
- Downloads whatever is currently in remote store
- If remote store is up-to-date (source uploaded before recovery started), this pre-populates the target
- Peer recovery phase1 then only needs to send files that differ
- `cleanFiles` at the end removes any stale files not in source metadata

**Post-recovery call (innerOpenEngineAndTranslog:3141)**:
- Runs AFTER `cleanFiles` has committed the peer-transferred segments
- At this point, the local store has the correct files from peer recovery
- Remote store should have the same or newer data
- `overrideLocal=false` means it only downloads files with DIFFERENT checksums
- If remote has the SAME files (normal case), nothing is downloaded — all skipped
- If remote has NEWER files (source did another upload after peer recovery snapshot), those get downloaded — this is correct behavior (catch up)

### Potential Issue: Remote store has OLDER files

**Scenario**: Source takes catalog snapshot for peer recovery, then remote store hasn't been updated yet (upload is async/delayed). The pre-recovery `syncSegmentsFromRemoteSegmentStore` downloads older files. Then peer recovery sends newer files. Then `cleanFiles` removes stale files. Then post-recovery `syncSegmentsFromRemoteSegmentStore` runs again — but by now the source has uploaded, so remote has current data.

**Mitigation**: The pre-recovery call is wrapped in try-catch and failures are swallowed. Even if it downloads stale files, `cleanFiles` will remove them. The post-recovery call uses `overrideLocal=false`, so it won't overwrite newer peer-transferred files with older remote files (checksums would differ, but the download would only happen if the local file doesn't exist or has a different checksum — and since peer recovery already placed the correct file, `localDirectoryContains` returns true).

**Edge case**: What if remote store has a file with the SAME name but DIFFERENT content (older version) and the peer-transferred file has a different checksum? `localDirectoryContains` compares the local checksum against the remote's stored checksum. If they match (peer-transferred file matches remote), skip. If they don't match (remote is stale), it would download the stale file, OVERWRITING the correct peer-transferred file.

**But**: This edge case is prevented by the recovery protocol. The source takes a catalog snapshot, which pins the files. The source's remote store upload happens from the same or newer snapshot. By the time the target opens its engine (post-recovery), the source has already uploaded the snapshot used for peer recovery. The remote metadata will have checksums matching the peer-transferred files.

### VERDICT: MITIGATED (with minor theoretical risk)

The system is designed correctly for the normal case:
- Pre-recovery sync is best-effort (exception swallowed)
- `cleanFiles` removes stale files after peer recovery
- Post-recovery sync uses `overrideLocal=false` and checksum comparison
- DFA checksum comparison works correctly via `DataFormatAwareStoreDirectory.calculateUploadChecksum()`

**Theoretical risk**: If the source's remote store upload is significantly delayed (source crashes after peer recovery but before uploading), the post-recovery sync could download stale files. However, this is not DFA-specific — it affects all remote-store-enabled indices equally. The existing comment in `PeerRecoveryTargetService` acknowledges this is a "temporary mitigation" pending a revamp (GitHub Issue #11331).
